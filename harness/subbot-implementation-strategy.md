# 실제 서브봇 구현 방식

## 목적

문서로 정의한 서브봇 체계를 실제 OpenClaw 운영 환경에서 어떻게 구현할지 기준을 정한다.

이 문서는 개념적 서브봇 정의를 실제 실행 가능한 구조로 전환하기 위한 구현 전략 문서다.

## 결론

현재 단계의 기본 구현 전략은 **하이브리드 방식**으로 한다.

- 기본은 마스터봇 중심 운영
- 필요할 때만 실제 서브봇 세션 생성
- 반복 감시나 분리 가치가 큰 작업만 유지 세션 검토

## 구현 방식 비교

### 1. 문서 기반 가상 서브봇

의미:
- 실제 별도 세션을 만들지 않고, 마스터봇이 정의된 역할과 템플릿에 따라 내부적으로 역할 전환해 처리한다.

장점:
- 구현이 가장 빠름
- 운영 복잡도가 낮음
- 초기 검증에 적합

단점:
- 진짜 멀티 에이전트 분리 효과는 약함
- 역할 경계가 흐려질 수 있음

### 2. 실제 분리 세션 서브봇

의미:
- 각 서브봇을 별도 세션 또는 서브에이전트로 생성해 실제 분리 운영한다.

장점:
- 역할 분리 검증에 유리함
- 병렬 또는 독립 작업에 적합
- 향후 확장에 유리함

단점:
- 구현과 운영 복잡도가 올라감
- 세션 관리와 비용 관리 필요

### 3. 하이브리드 방식

의미:
- 평소에는 마스터봇 중심으로 운영하되, 특정 작업에서만 실제 서브봇 세션을 생성한다.

장점:
- 현재 단계에서 가장 현실적
- 설계 검증과 실제 분리 검증을 동시에 할 수 있음
- 비용과 복잡도를 통제하기 좋음

단점:
- 어떤 경우에 실제 세션을 띄울지 기준이 필요함

## 권장 구현 전략

### 기본 원칙

- 기본 응답과 간단한 작업은 마스터봇이 직접 처리
- 역할 특화가 실제로 필요한 작업만 서브봇 세션 생성
- 워치봇처럼 장기 감시 성격이 큰 경우만 유지 세션 검토
- 세션 지속성보다 기록과 문서를 우선 기준으로 사용

## 실제 세션 생성 권장 대상

### 우선 실제 세션 생성 검증 대상

- 리서치봇
- 빌더봇
- 리뷰봇

이유:
- 연쇄 위임 테스트가 명확함
- 각 역할의 차이가 분명함
- 실제 분리 검증 가치가 큼

### 초기에는 굳이 분리하지 않아도 되는 대상

- 메모리봇
- 스케줄봇
- 태스크봇

이유:
- 비교적 짧고 단일 흐름 작업이 많음
- 먼저 문서 기반 또는 마스터봇 보조 방식으로도 검증 가능

### 예외적으로 장기 운영 후보

- 워치봇

이유:
- 감시 성격상 유지 세션 또는 반복 실행형이 어울림

## 구현 단계 제안

### 1단계. 하이브리드 기준 확정

- 실제 세션을 띄울 봇과 띄우지 않을 봇 구분
- 세션 생성 기준 확정

### 2단계. 서브봇 호출 포맷 확정

- 실제 세션 생성 시 사용할 프롬프트/입력 패킷 형식 확정

### 3단계. 우선 1개 봇 실구현

- 리서치봇 또는 빌더봇 하나를 실제 서브봇 세션으로 구현

### 4단계. 연쇄 구현

- 리서치봇 → 빌더봇 → 리뷰봇 흐름 중 일부 또는 전체 구현

### 5단계. 시범 테스트

- 실제 요청을 넣어 마스터봇이 위임하고 결과를 통합하는지 확인

## 운영 결론

현재는 모든 서브봇을 상시 독립 운영하는 단계가 아니라, 마스터봇 중심 구조 위에 필요한 순간만 실제 서브봇을 생성하는 하이브리드 방식이 가장 적절하다. 이후에는 리서치봇, 빌더봇, 리뷰봇부터 실제 세션 기반 구현을 시작하는 것이 좋다.
