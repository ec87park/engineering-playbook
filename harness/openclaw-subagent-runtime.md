# OpenClaw 서브에이전트 런타임 방식

## 목적

OpenClaw 안에서 마스터봇과 별도로 서브봇을 어떻게 실행하는지 이해할 수 있도록, 실제 서브에이전트 세션 방식의 개념과 동작 구조를 설명한다.

## 핵심 설명

서브봇을 "실제로 구현한다"는 것은 OpenClaw를 하나 더 설치하는 뜻이 아니다.
같은 OpenClaw 환경 안에서 **분리된 작업 세션**을 하나 더 생성해, 그 세션에 특정 역할만 맡기는 방식이다.

즉:
- 현재 대화 세션 = 마스터봇 세션
- 추가로 생성하는 작업 세션 = 서브봇 세션

## 어떻게 동작하는가

### 1. 마스터봇이 작업을 판단한다

예:
- 이 요청은 조사 역할이 필요하다고 판단

### 2. OpenClaw 안에서 새 세션을 생성한다

- 현재 세션과 분리된 작업용 세션이 만들어진다.
- 이 세션은 같은 워크스페이스를 공유할 수 있다.
- 하지만 대화 문맥과 역할은 분리된다.

### 3. 새 세션에 역할을 부여한다

예:
- "당신은 리서치봇이다"
- "이 범위만 조사하라"
- "이 형식으로 결과를 반환하라"

### 4. 서브봇 세션이 작업한다

- 조사, 작성, 검토 같은 역할별 작업 수행
- 필요하면 파일 읽기, 웹 검색 등 사용

### 5. 결과를 마스터봇이 회수한다

- 서브봇 세션의 결과를 받아
- 최종 사용자 응답으로 재구성한다.

## 왜 분리 세션을 쓰는가

- 역할 혼선을 줄이기 위해
- 작업 문맥을 분리하기 위해
- 특정 작업만 집중시켜 품질을 높이기 위해
- 마스터봇은 통합과 판단에 집중하기 위해

## 설치가 더 필요한가

아니다.

- OpenClaw를 하나 더 설치할 필요 없음
- 별도 서버를 띄울 필요 없음
- 같은 OpenClaw 안에서 세션만 추가 생성하는 구조

## 구현 관점 정리

실제 구현은 다음과 같은 흐름이다.

1. 마스터봇이 서브봇 필요 판단
2. OpenClaw에서 서브세션 생성
3. 호출 포맷에 맞는 역할 지시 전달
4. 서브봇 결과 수신
5. 마스터봇이 결과 통합

## 현재 적용 방향

현재는 하이브리드 전략을 기준으로 한다.

- 기본은 마스터봇 중심 운영
- 필요할 때만 실제 서브에이전트 세션 생성
- 우선 구현 대상은 리서치봇, 빌더봇, 리뷰봇

## 운영 결론

OpenClaw에서 서브봇 구현은 시스템을 하나 더 설치하는 일이 아니라, 같은 환경 안에서 역할이 분리된 작업 세션을 추가로 생성하는 일이다. 핵심은 설치가 아니라 역할 분리와 작업 흐름 설계에 있다.
