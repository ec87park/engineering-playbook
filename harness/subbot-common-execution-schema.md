# 서브봇 공통 실행 스키마

## 목적

마스터봇과 서브봇이 일관된 방식으로 작업을 주고받을 수 있도록 공통 실행 스키마를 정의한다.

이 문서는 개념 정의 수준을 넘어, 실제 위임과 응답이 어떤 구조를 따라야 하는지를 정리한다.

## 적용 범위

이 스키마는 다음 대상에 공통 적용한다.

- 마스터봇이 서브봇에 작업을 위임할 때
- 서브봇이 작업 결과를 반환할 때
- 실패, 보류, 에스컬레이션을 보고할 때

## 핵심 원칙

### 1. 모든 위임은 구조화한다

작업은 자연어 한 줄로 던지지 않고, 최소한 목적, 범위, 출력 기대치를 포함한 구조로 전달한다.

### 2. 모든 결과는 재조합 가능해야 한다

서브봇의 응답은 마스터봇이 그대로 재사용하거나 재편집할 수 있어야 한다.

### 3. 실패와 보류도 표준 형식으로 보고한다

막혔을 때는 침묵하거나 임의 판단하지 않고, 표준화된 형태로 이유와 필요한 지원을 보고한다.

### 4. 문맥은 필요한 만큼만 전달한다

모든 대화 기록을 통째로 넘기지 않고, 현재 작업에 필요한 핵심 문맥만 포함한다.

## 1. 마스터봇 입력 스키마

마스터봇은 사용자 요청을 받으면 내부적으로 아래 항목으로 정리한다.

- 요청 요약
- 목적
- 제약 조건
- 우선순위
- 위험도
- 관련 문맥
- 기대 산출물

예시 항목:

- 요청 요약: 이번 주 일정 정리
- 목적: 중요한 일정과 비는 시간 파악
- 제약 조건: 오늘 안에 요약 필요
- 우선순위: 높음
- 위험도: 낮음
- 관련 문맥: 기존 일정 관리 기준
- 기대 산출물: 오늘/이번 주 일정 요약

## 2. 서브봇 위임 스키마

마스터봇이 서브봇에 넘기는 공통 구조는 아래와 같다.

### 필수 항목

- 작업 ID
- 대상 봇
- 작업 목적
- 요청 내용
- 작업 범위
- 입력 문맥
- 기대 출력 형식
- 완료 조건
- 제약 사항
- 에스컬레이션 조건

### 설명

- 작업 ID: 추적 가능한 식별자
- 대상 봇: 어떤 서브봇인지 명시
- 작업 목적: 이 작업이 왜 필요한지
- 요청 내용: 실제 수행해야 할 일
- 작업 범위: 어디까지 다루는지
- 입력 문맥: 관련 문서, 기존 기준, 사용자 선호 등
- 기대 출력 형식: 요약, 비교표, 초안, 검토 리포트 등
- 완료 조건: 무엇이 충족되면 완료인지
- 제약 사항: 금지사항, 승인 필요 조건, 시간 제한 등
- 에스컬레이션 조건: 언제 마스터봇으로 되돌릴지

## 3. 서브봇 출력 스키마

서브봇은 작업 완료 시 아래 구조로 반환한다.

### 기본 출력 항목

- 상태
- 결과 요약
- 상세 결과
- 근거 또는 판단 기준
- 남은 리스크
- 후속 제안

### 상태 값 예시

- 완료
- 부분 완료
- 보류
- 실패
- 에스컬레이션 필요

### 설명

- 상태: 현재 결과 상태
- 결과 요약: 한눈에 보는 핵심
- 상세 결과: 실제 산출물 내용
- 근거 또는 판단 기준: 왜 이렇게 판단했는지
- 남은 리스크: 확정 전 주의할 점
- 후속 제안: 다음에 이어질 작업

## 4. 실패 및 보류 스키마

서브봇이 막혔을 때는 아래 항목으로 보고한다.

- 상태: 보류 또는 실패
- 막힌 이유
- 현재까지 진행한 것
- 마스터봇에 필요한 지원
- 계속 진행 가능 여부

예:
- 상태: 에스컬레이션 필요
- 막힌 이유: 요구 범위 불명확
- 현재까지 진행한 것: 초안 구조 작성
- 필요한 지원: 우선순위 확정
- 계속 진행 가능 여부: 일부만 가능

## 5. 에스컬레이션 스키마

아래 경우에는 마스터봇으로 되돌린다.

- 요구사항이 모호함
- 권한 또는 승인 부족
- 다른 봇과 충돌 발생
- 결과 신뢰도가 낮음
- 범위 밖 요청이 들어옴

에스컬레이션 시 포함 항목:
- 무엇이 문제인지
- 왜 단독 처리 불가인지
- 어떤 선택지가 있는지
- 마스터봇이 결정해야 할 사항

## 6. 공통 출력 형식 원칙

모든 서브봇 응답은 다음 조건을 만족해야 한다.

- 마스터봇이 바로 이해 가능해야 한다.
- 다시 편집하거나 합치기 쉬워야 한다.
- 불필요하게 장황하지 않아야 한다.
- 결과와 판단 근거가 구분되어야 한다.

## 7. 마스터봇 재조합 규칙

마스터봇은 서브봇 결과를 받을 때 아래 순서로 처리한다.

1. 상태 확인
2. 결과 요약 확인
3. 리스크 및 누락 확인
4. 다른 서브봇 결과와 충돌 여부 확인
5. 사용자용 최종 응답으로 재조합

## 8. 운영 결론

서브봇 시스템이 실제로 작동하려면, 좋은 역할 정의만으로는 부족하다. 모든 봇은 구조화된 입력을 받고, 구조화된 출력을 반환하며, 실패와 에스컬레이션도 같은 규격 안에서 처리해야 한다. 이 문서는 그 실행 규격의 공통 기반이다.
