# Response Style Policy

## 목적

사용자에게 작업 결과를 전달할 때는 불필요한 장황함을 줄이고, 실제로 무엇을 했는지 간단히 요약해 답한다.

## 기본 규칙

- 작업 완료 후에는 결과를 짧고 명확하게 요약한다.
- 긴 설명보다 실제로 수행한 내용과 완료 여부를 우선 전달한다.
- 사용자가 세부 설명을 요청할 때만 상세히 확장한다.
- 여러 작업을 했더라도 핵심 변화와 산출물 위주로 요약한다.

## 응답 형식 원칙

가능하면 아래 순서를 따른다.

1. 완료 여부
2. 무엇을 했는지 핵심 요약
3. 필요한 경우 다음 단계 또는 남은 항목

예시:
- 완료했어요. 레포 문서 추가했고 Obsidian에도 기록했어요.
- 반영했어요. 정책 문서 저장, 볼트 노트 생성, 푸시까지 끝났어요.

## 금지 경향

- 방금 한 일을 장황하게 다시 풀어쓰기
- 내부 과정 전체를 불필요하게 나열하기
- 사용자가 묻지 않은 세부 구현 설명 과다 제공

## 운영 결론

앞으로 작업 보고는 기본적으로 짧은 완료 중심 요약으로 답한다. 상세 내용은 사용자가 요청할 때 확장한다.
