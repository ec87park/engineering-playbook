# Context Hygiene Policy

## 목적

모든 봇은 장기 세션 운영 중 발생할 수 있는 컨텍스트 오염, 기준 드리프트, 최근 문맥 편향을 줄이기 위해 기록 중심 운영과 주기적 정리를 수행해야 한다.

이 문서는 마스터봇과 모든 서브봇이 따라야 하는 공통 컨텍스트 위생 정책을 정의한다.

## 정책 원칙

### 1. 대화는 작업 공간이고, 기록은 기준 원본이다

세션 대화는 임시 작업 공간으로 취급한다.
중요한 기준, 결정, 사용자 선호, 프로젝트 구조, 운영 규칙은 반드시 파일 기반 기록으로 고정한다.

### 2. 중요한 합의는 즉시 외부화한다

다음 성격의 내용은 가능한 한 빠르게 레포 문서 또는 Obsidian 노트로 외부화한다.

- 운영 기준
- 역할 정의
- 승인 규칙
- 프로젝트 결정사항
- 사용자 선호
- 반복적으로 참조될 내용

### 3. 문서가 기억보다 우선한다

봇은 자신의 대화 기억보다 기록된 문서를 우선 참조해야 한다.
문서와 세션 기억이 충돌하면 문서를 기준 진실원본으로 간주한다.

### 4. 세션 길이는 무제한 자산이 아니다

컨텍스트가 크더라도 장기 세션은 드리프트와 오염을 유발할 수 있다.
따라서 모든 봇은 일정 범위 이상의 누적 문맥에서 압축, 정리, 재앵커링, 초기화를 고려해야 한다.

## 공통 규칙

### 규칙 1. 상시 기록

모든 봇은 작업 중 중요한 판단과 결과를 적절한 위치에 기록할 수 있어야 한다.

- 시스템 규칙, 구조, 역할, 정책 → engineering-playbook 레포
- 개인 기록, 운영 메모, 장기 참조 노트 → Obsidian vault

### 규칙 2. 세션 내 암묵 합의 금지

문서화되지 않은 중요한 합의를 장기 기준처럼 다루지 않는다.
중요한 기준은 반드시 외부 기록으로 남긴다.

### 규칙 3. 최근 문맥 편향 경계

최근 메시지가 기존 기준과 충돌할 경우, 봇은 즉시 기존 문서를 재확인해야 한다.

### 규칙 4. 요약과 재앵커링

세션이 길어질수록 봇은 다음을 수행해야 한다.

- 지금까지의 핵심 기준 요약
- 문서 기준 재확인
- 불필요한 대화 문맥 의존 축소
- 다음 단계에 필요한 핵심 정보만 유지

### 규칙 5. 필요한 경우 초기화 허용

세션 품질이 저하되거나 기준 충돌 가능성이 커지면, 새로운 세션으로 전환하고 문서를 기준으로 다시 시작할 수 있다.

## 컨텍스트 관리 임계 규칙

정확한 수치는 모델과 런타임에 따라 달라질 수 있으므로 절대값보다 운영 기준을 우선한다. 다만 아래 기준을 기본선으로 둔다.

### 1. 주의 구간

다음 중 하나에 해당하면 컨텍스트 정리 필요성을 검토한다.

- 긴 설계 논의가 여러 주제에 걸쳐 누적됨
- 최근 메시지가 초기 합의를 덮기 시작함
- 동일한 규칙을 다시 정의하는 일이 반복됨
- 세션 사용량이 전체 한도의 절반을 넘기기 시작함

### 2. 압축 구간

다음 중 하나에 해당하면 요약과 외부 기록 갱신을 우선 수행한다.

- 세션 사용량이 전체 한도의 약 60% 이상
- 주요 규칙과 역할 정의가 다수 누적됨
- 프로젝트별 결정사항이 혼재되기 시작함

이 구간에서 봇은 아래를 수행한다.
- 핵심 합의를 문서화
- 이후 필요한 핵심만 요약
- 중복 논의를 줄이고 문서 참조 기반으로 전환

### 3. 재시작 구간

다음 중 하나에 해당하면 새 세션 시작을 적극 고려한다.

- 세션 사용량이 전체 한도의 약 75% 이상
- 기준 충돌 또는 응답 품질 저하가 감지됨
- 여러 프로젝트와 규칙이 한 세션에 과도하게 섞임
- 동일 질문에 일관성 없는 판단이 나타남

이 경우 봇은 아래를 수행한다.
- 핵심 기준을 레포 또는 Obsidian에 고정
- 짧은 운영 요약 작성
- 새 세션에서 문서를 우선 참조해 재개

## 봇별 의무

### 마스터봇
- 모든 중요한 합의를 문서로 외부화하도록 관리
- 세션 드리프트를 감지하면 재앵커링 또는 새 세션 전환 제안
- 문서 기준을 최종 판단의 우선순위로 유지

### 메모리봇
- Daily 기록 중 장기 가치가 있는 내용을 승격
- 중복, 오염, 오래된 내용을 정리
- 장기 기준과 임시 로그를 구분

### 워치봇
- 장시간 누적 세션보다 핵심 상태 변화만 보고
- 지속 로그가 필요하면 요약 중심으로 남김

### 스케줄봇, 태스크봇
- 당일 운영 문맥과 장기 기준을 분리
- 일정과 할 일 변경 기준은 문서로 고정

### 리서치봇, 빌더봇, 리뷰봇
- 작업 결과는 문서로 남기고 세션 의존도를 낮춤
- 조사 초안, 빌드 결과, 리뷰 포인트를 구조화된 산출물로 남김

## 하네스 제약 적용 원칙

모든 봇 하네스는 아래 제약을 포함해야 한다.

- 중요한 합의는 파일에 기록할 것
- 문서와 세션 기억이 충돌하면 문서를 우선할 것
- 일정 수준 이상 컨텍스트가 누적되면 요약, 압축, 재앵커링을 수행할 것
- 품질 저하 또는 혼선이 감지되면 새 세션 전환을 허용할 것

## 운영 결론

모든 봇은 긴 대화를 그대로 끌고 가는 방식이 아니라, 기록 중심 운영과 문서 우선 참조를 통해 컨텍스트 오염을 줄여야 한다. 장기 운영에서 기억의 신뢰성은 세션 길이가 아니라 외부 기록의 품질과 재앵커링 규칙에서 나온다.
