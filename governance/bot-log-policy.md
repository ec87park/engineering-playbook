# 봇 로그 기록 정책

## 목적

마스터봇과 각 서브봇이 주고받은 작업 요청과 응답을 추적 가능하게 남겨, 운영 흐름을 복기하고 품질을 개선할 수 있도록 로그 기록 정책을 정의한다.

## 기본 원칙

- 중요한 위임과 응답은 기록으로 남긴다.
- 기록은 대화 원문 전체를 무조건 복사하기보다, 운영상 의미 있는 수준으로 구조화해 남긴다.
- 마스터봇 요청, 서브봇 응답, 후속 판단을 분리해 기록한다.
- 민감 정보와 불필요한 장문 원문은 요약 중심으로 줄인다.

## 볼트 저장 구조

봇 간 작업 로그는 Obsidian vault의 아래 경로에 저장한다.

- `31 Bot Logs/Masterbot/`
- `31 Bot Logs/Memorybot/`
- `31 Bot Logs/Watchbot/`
- `31 Bot Logs/Schedulebot/`
- `31 Bot Logs/Taskbot/`
- `31 Bot Logs/Researchbot/`
- `31 Bot Logs/Builderbot/`
- `31 Bot Logs/Reviewbot/`

## 기록 대상

다음 항목은 기록 우선 대상이다.

- 실제 서브봇 세션 호출
- 역할이 분명한 위임 작업
- 연쇄 위임 결과
- 실패, 보류, 에스컬레이션 사례
- 운영 규칙 개선에 참고할 만한 상호작용

## 기록 형식

각 로그는 가능하면 아래 항목을 포함한다.

- 날짜/시각
- 작업 ID 또는 구분 이름
- 호출 주체
- 대상 봇
- 요청 요약
- 응답 요약
- 결과 상태
- 후속 판단 또는 후속 작업

## 기록 수준

### 1. 짧은 로그

일반적인 성공 사례는 아래 정도로 남긴다.
- 무엇을 요청했는지
- 무엇을 돌려받았는지
- 결과가 어떻게 활용됐는지

### 2. 확장 로그

다음 경우는 더 자세히 남긴다.
- 실패
- 보류
- 에스컬레이션
- 기준 충돌
- 중요한 설계 판단

## 운영 규칙

- 실제 서브봇 세션을 생성한 경우 로그 기록을 기본으로 한다.
- 마스터봇은 서브봇 호출 후 요약 로그를 남긴다.
- 서브봇 결과가 중요한 판단에 쓰였으면 결과 요약과 후속 판단을 함께 남긴다.
- 같은 유형의 반복 로그는 필요 시 일일 요약 형태로 묶을 수 있다.

## 로그 작성 예시

- 요청: 하이브리드 서브봇 구현 방식 장단점 조사
- 응답: 하이브리드 방식의 장점, 리스크, 적합 상황 구조화 반환
- 상태: 완료
- 후속: 세션 생성 기준 체크리스트 문서화 필요

## 운영 결론

서브봇 시스템은 결과물뿐 아니라 위임과 응답의 흐름도 자산이다. 따라서 마스터봇과 서브봇의 주요 상호작용은 볼트의 전용 로그 공간에 구조화해 남기고, 이후 운영 개선의 근거로 활용한다.
