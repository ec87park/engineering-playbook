# 리서치봇 파일럿 검토

## 목적

리서치봇 파일럿 실행 결과를 정리하고, 실제 구현 단계에서 무엇을 유지하고 무엇을 보완할지 결정하기 위한 검토 문서다.

## 이번 파일럿에서 확인된 점

### 1. 하이브리드 방식은 초기 운영안으로 현실적이다

이번 파일럿 기준으로는 모든 역할을 항상 분리 세션으로 운영하기보다, 마스터봇 중심으로 처리하되 필요한 경우에만 실제 서브봇 세션을 생성하는 방식이 가장 현실적이었다.

확인된 이유:
- 초기 운영 복잡도를 낮출 수 있다.
- 역할 분리 검증이 필요한 구간만 선택적으로 실험할 수 있다.
- 문서 기반 운영 기준과 실제 세션 기반 검증을 함께 가져갈 수 있다.

### 2. 리서치봇은 실제 세션 분리 가치가 높다

리서치봇은 조사 근거 정리, 비교 기준 유지, 불확실성 표시가 중요해 마스터봇 내부 처리보다 별도 세션으로 분리했을 때 역할 경계가 더 분명해진다.

특히 아래 상황에서 분리 가치가 컸다.
- 후속 빌더봇 문서 작성의 입력이 되는 조사 결과를 만들 때
- 비교 항목과 판단 근거를 구조적으로 남겨야 할 때
- 불확실한 내용을 명시적으로 분리해 다뤄야 할 때

### 3. 세션 생성 기준 체크리스트가 핵심 운영 장치다

하이브리드 방식이 실용적이려면, 언제 실제 세션을 만들지 일관되게 판단할 기준이 필요하다.
이번 파일럿에서는 이 기준이 모호하면 불필요한 위임이나 역할 혼선이 생길 수 있다는 점이 분명해졌다.

따라서 리서치봇 운영에서는 아래 항목을 먼저 확인해야 한다.
- 역할 특화가 실제로 필요한가
- 결과가 후속 작업에 재사용되는가
- 작업 범위와 종료 조건이 명확한가
- 문맥 분리 이점이 충분한가

## 남은 리스크

### 1. 세션 생성 판단이 사람 감각에 의존할 수 있음

체크리스트가 있어도 실제 적용 기준이 충분히 예시화되지 않으면, 같은 유형의 작업에서도 세션 생성 여부가 흔들릴 수 있다.

### 2. 조사 출력의 품질 편차 가능성

리서치봇 출력 형식이 고정되지 않으면, 후속 빌더봇이나 리뷰봇이 입력으로 쓰기 어려운 결과가 나올 수 있다.

### 3. 연쇄 위임 시 문맥 손실 가능성

리서치봇 결과를 빌더봇으로 넘길 때, 핵심 판단 근거와 불확실성 정보가 축약되거나 빠질 위험이 있다.

## 다음 단계 제안

### 1. 리서치봇 세션 생성 기준을 예시 중심으로 보강

현재 체크리스트를 유지하되,
- 직접 처리할 요청 예시
- 단기 세션 생성이 필요한 요청 예시
를 함께 붙여 실제 판단 일관성을 높인다.

### 2. 리서치봇 출력 최소 형식을 고정

리서치봇 결과는 최소한 아래 구조를 유지하도록 맞춘다.
- 상태
- 결과 요약
- 핵심 근거 또는 비교 결과
- 남은 리스크
- 후속 제안

### 3. 리서치봇 → 빌더봇 연쇄 흐름을 작은 범위로 재검증

다음 구현 단계에서는 리서치 결과가 실제 문서 초안 작성으로 자연스럽게 이어지는지 작은 시나리오로 다시 확인한다.
이 단계에서 호출 포맷, 출력 구조, 문맥 전달 방식의 부족한 부분을 함께 보완한다.

## 운영 판단

현재 단계에서는 리서치봇을 상시 분리 운영하기보다, 하이브리드 방식을 유지하면서 실제 세션 생성 기준과 출력 형식을 먼저 안정화하는 것이 적절하다.
다음 구현 초점은 "더 많이 분리하는 것"이 아니라 "언제 분리할지와 분리했을 때 무엇을 남길지"를 명확히 하는 데 둔다.
