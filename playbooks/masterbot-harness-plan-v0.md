# 마스터봇 하네스 엔지니어링 기획 초안

## 목표 정의

우리가 만들려는 것은 단순한 봇 묶음이 아니라 다음을 수행하는 체계다.

- 마스터봇이 사용자 요청을 해석한다.
- 적절한 서브봇을 만들거나 호출한다.
- 각 봇의 역할, 권한, 금지사항, 출력 형식을 통제한다.
- 최종 결과를 검수해서 사용자에게 전달한다.

즉, 전체 방향은 "멀티 에이전트 운영체계"에 가깝다.

## 큰 구조

### A. 마스터봇

역할:
- 사용자 요청 해석
- 작업 분해
- 필요한 서브봇 결정
- 서브봇별 규칙 및 권한 주입
- 결과 취합 및 품질 관리
- 장기 메모리와 운영 정책 관리

### B. 서브봇

역할:
- 특정 업무만 수행
- 좁은 권한과 명확한 출력 계약을 가짐
- 필요시 생성하고, 반복 업무는 고정 운영 가능

예시:
- 리서치봇
- 개발봇
- 문서봇
- 일정/알림봇
- QA봇
- 모니터링봇

### C. 하네스 레이어

서브봇 자체보다 중요한 것은 어떻게 안전하게 실행할지를 관리하는 운영 레이어다.

하네스가 관리할 항목:
- 시스템 프롬프트 템플릿
- 역할 정의
- 허용 도구
- 금지 행위
- 승인 필요 조건
- 출력 포맷
- 실패 처리 규칙
- 로그 및 감사
- 메모리 범위
- 세션 수명주기

## 하네스 엔지니어링 관점의 핵심 축

최소 6축으로 설계한다.

### 1) Role

각 봇은 무엇을 하는가?
- 책임
- 비책임
- 성공 기준

예:
- 개발봇: 코드 수정, 테스트, 커밋
- 비책임: 사용자에게 직접 정책 확정 답변
- 성공 기준: 테스트 통과, 변경 요약, 리스크 명시

### 2) Guardrails

무엇을 하면 안 되는가?
- 외부 발신 금지
- destructive action 금지
- 비밀정보 노출 금지
- 사용자 승인 전 배포 금지

### 3) Tools

무슨 도구를 쓸 수 있는가?
- read/write/edit 가능 여부
- exec 가능 여부
- web_search 가능 여부
- 세션 간 메시징 가능 여부

### 4) IO Contract

입력과 출력 형식.
- 입력 스키마
- 출력 스키마
- 에러 보고 형식
- 중간 진행 보고 규칙

### 5) Memory Scope

무엇을 기억하거나 참조 가능한가?
- 공용 정책 메모리
- 프로젝트 메모리
- 민감 메모리 차단
- 세션 간 공유 범위

### 6) Escalation

언제 마스터봇으로 다시 올리는가?
- 요구사항이 모호함
- 권한 부족
- 위험한 명령
- 결과 신뢰도 낮음
- 정책 충돌

## 운영 원칙 초안

마스터봇 체계의 운영 헌장 성격으로 관리한다.

- 마스터봇만 사용자와 최종 계약을 맺는다.
- 서브봇은 위임된 범위 안에서만 행동한다.
- 서브봇은 정책 변경 권한이 없다.
- 위험 작업은 승인 게이트를 통과해야 한다.
- 모든 중요한 결정은 로그나 메모리에 남긴다.
- 최종 응답 전 마스터봇이 검수한다.

## 추천 아키텍처

처음부터 과도하게 복잡하게 가지 말고 3계층으로 시작한다.

### 레벨 1. 정책 계층

예시 파일:
- `governance/constitution.md`
- `governance/policies.md`
- `governance/risk-matrix.md`

### 레벨 2. 에이전트 정의 계층

예시 파일:
- `agents/masterbot.md`
- `agents/subbots/researcher.md`
- `agents/subbots/developer.md`
- `agents/subbots/reviewer.md`

각 파일에는 다음을 포함한다.
- 정체성
- 역할
- 입력
- 출력
- 허용 도구
- 금지사항
- escalation 규칙

### 레벨 3. 실행 하네스 계층

예시 파일:
- `harness/templates/`
- `harness/checklists/`
- `harness/workflows/`
- `harness/session-rules.md`

여기서 정의할 항목:
- 프롬프트 조립
- 세션 생성 규칙
- 태스크 분배 규칙
- 종료 규칙
- 결과 검수 규칙

## 처음 만들 서브봇 추천

처음에는 3개로 시작한다.

### 1) Research Bot
- 조사
- 자료 요약
- 옵션 비교
- 근거 수집

### 2) Builder Bot
- 코드, 문서, 설정 생성
- 구현안 작성
- 수정 작업

### 3) Reviewer Bot
- 결과 검토
- 누락, 리스크, 테스트 체크
- 마스터봇에게 검수 리포트 제공

이 세 종류만으로도 대부분의 프로젝트를 처리할 수 있다.

## 하네스 문서 템플릿 예시

각 서브봇 정의는 다음 항목으로 통일한다.

- 이름
- 목적
- 담당 범위
- 하지 말아야 할 일
- 허용 도구
- 입력 형식
- 출력 형식
- 실패 시 행동
- 마스터봇으로 에스컬레이션 조건
- 예시 작업

## 지금 단계에서 가장 먼저 해야 할 일

코딩보다 먼저 아래 3가지를 정한다.

### 1) 운영 철학
- 마스터봇을 어디까지 중앙집권적으로 관리할지
- 서브봇 자율성을 어디까지 줄지

### 2) 위험 등급
- low / medium / high risk 작업 분류
- 승인 필요 규칙

### 3) 기본 서브봇 카탈로그
- 어떤 역할의 서브봇을 표준으로 둘지

## 추천 진행 순서

1. 마스터봇 운영 철학 문서 작성
2. 하네스 공통 스키마 정의
3. 서브봇 3종 초안 정의
4. 승인 및 에스컬레이션 규칙 정의
5. 실제 작은 프로젝트에 시범 적용
6. 운영하면서 수정

## 추천 결론

이 방향은 단순히 프롬프트를 다듬는 수준이 아니라 "에이전트 조직 운영 체계"를 만드는 작업이다.

권장 순서:
- 헌장(원칙)
- 에이전트 스펙
- 실행 하네스
- 자동화

순서가 뒤집히면 봇이 늘수록 관리 복잡도가 빠르게 커질 수 있다.
